# F-006 撤销后重新记录设计

**创建时间：** 2026-04-26
**功能ID：** F-006
**状态：** 设计完成

---

## 一、概述

**功能名称：** 撤销后重新记录
**功能描述：** 用户保存训练记录后，可在消息气泡内撤销，撤销后输入框保留上下文，用户可直接修改重新发送。

**优先级：** P0（L1 基础功能层）
**依赖关系：** F-004 一键记录模式、F-001 视觉卡片

---

## 二、核心交互

### 2.1 撤销入口

**位置：** 用户消息气泡内
**触发：** 鼠标悬停气泡，显示撤销图标（🔄 或 ⟲）
**显示：** 默认隐藏，hover 时显示

### 2.2 撤销行为

1. 用户点击撤销图标
2. 后端软删除该记录（deleted_at 更新）
3. 前端更新消息气泡样式
4. 输入框保留上下文

### 2.3 撤销后重新记录

- 输入框保留撤销前的输入内容
- 用户可直接修改或使用语音重新记录
- 无需重新输入完整内容

---

## 三、视觉设计

### 3.1 正常状态

```
┌────────────────────────────────────┐
│ 深蹲100公斤5组8个              🔄  │  ← 悬停显示撤销图标
└────────────────────────────────────┘
```

### 3.2 撤销后状态

```
┌────────────────────────────────────┤
│ ~~深蹲100公斤5组8个~~        [已撤销] │  ← 灰色 + 删除线
└────────────────────────────────────┘
```

**样式：**
- 文字颜色：#94a3b8（灰色）
- 删除线：text-decoration: line-through
- 标签：右侧显示"已撤销"标签

---

## 四、数据处理

### 4.1 软删除

- 调用现有 DELETE /api/records/workout/:id
- 后端设置 deleted_at 为当前时间
- 数据保留，可通过 restore 接口恢复

### 4.2 恢复接口

- POST /api/records/workout/:id/restore
- 清除 deleted_at，恢复数据显示

---

## 五、接口契约

### 5.1 撤销请求

```
DELETE /api/records/workout/:id
Authorization: Bearer <jwt>
```

### 5.2 撤销响应

```json
{
  "success": true,
  "message": "记录已撤销"
}
```

### 5.3 恢复请求

```
POST /api/records/workout/:id/restore
Authorization: Bearer <jwt>
```

### 5.4 恢复响应

```json
{
  "success": true,
  "message": "记录已恢复"
}
```

## 六、状态管理

| 状态 | 样式 | 说明 |
|------|------|------|
| normal | 正常显示 | 可点击撤销 |
| revoked | 灰色 + 删除线 + "已撤销"标签 | 不可撤销 |
| restoring | 显示恢复中 | 等待后端响应 |
| restored | 恢复正常显示 | 恢复成功 |

---

## 七、上下文保留

### 7.1 保留内容

撤销后，输入框保留：
- 用户最后发送的消息内容
- 语音转文字后的原始文本

### 7.2 示例

```
用户发送："深蹲100公斤5组8个" → 保存成功 → 撤销 → 输入框保留"深蹲100公斤5组8个"
用户修改："深蹲120公斤5组8个" → 重新发送
```

---

## 八、与 F-004/F-001 的配合

```
F-004 语音记录 → 保存成功 → 显示 F-001 视觉卡片
     ↓
用户撤销 → 输入框保留上下文 → 用户修改内容 → 重新发送
```

---

## 九、前端实现要点

1. **撤销图标：** 消息气泡 hover 时显示，点击触发撤销
2. **样式变更：** 撤销成功后更新气泡样式（灰色 + 删除线）
3. **输入框保留：** 撤销后不清空输入框
4. **乐观更新：** 前端先更新 UI，后端异步删除
